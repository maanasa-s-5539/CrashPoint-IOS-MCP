# CrashPoint iOS MCP — Setup & Execution Guide

For **Developers** and **QA** who need to set up, configure, and run the CrashPoint iOS MCP server end-to-end.

---

## ⚡ Two Execution Paths — Read This First

There are **two ways** to run the CrashPoint pipeline. They have **different setup requirements**:

| | **Path A: Claude Desktop / Cursor (MCP tools)** | **Path B: Terminal bash script** |
|---|---|---|
| **First-time setup** | None — workspace is auto-created on the first `run_full_pipeline` call | One-time: `CRASH_ANALYSIS_PARENT=... node dist/cli.js setup` (generates the bash script with real paths) |
| **How to run** | Ask Claude to call `run_full_pipeline` or `run_basic_pipeline` | `bash /path/to/ParentHolderFolder/Automation/run_crash_pipeline.sh` |
| **Bash script generated?** | ✅ Yes — auto-generated during first-run setup | ✅ Yes — generated by the `setup` command |
| **Sections that apply** | 1, 2, 3, 5, 8, 9 | 1, 2, 3, 4, 5, 6, 7, 8, 9 |

**Key message:** The MCP tool path is **fully automatic** — just configure your MCP client and start using it. The terminal bash script path requires **one** manual setup call first because the bash script itself doesn't exist until `node dist/cli.js setup` creates it by replacing `<REPLACE_WITH_...>` placeholders with your real paths.

## 1. Prerequisites

```bash
# Node.js 18 or higher
node --version

# Xcode Command Line Tools (required for symbolication)
xcode-select --install
xcrun --find symbolicatecrash

# jq (required for the automated bash pipeline)
brew install jq
```

## 3. Create the Config File

Choose a folder that will hold all crash data — this is your **ParentHolderFolder**. Copy the example config into it:

```bash
cp crashpoint.config.example.json /path/to/ParentHolderFolder/crashpoint.config.json
```

Open `crashpoint.config.json` and fill in your values:

**Where to find key values:**

| Key | Where to find it |
|---|---|
| `CRASH_ANALYSIS_PARENT` | The absolute path to your ParentHolderFolder |
| `DSYM_PATH` | In your Xcode archive: right-click archive → "Show in Finder" → `.xcarchive/dSYMs/MyApp.app.dSYM` |
| `APP_PATH` | Same archive: `.xcarchive/Products/Applications/MyApp.app` |
| `APP_NAME` | The filename without `.app`, e.g. `MyApp` |
| `MASTER_BRANCH_PATH` | Wherever you have your master branch checked out locally |
| `DEV_BRANCH_PATH` | Wherever you have your development branch checked out locally |
| `CLAUDE_CLI_PATH` | Run `which claude` in Terminal after installing Claude CLI |

**Crash window date settings:**

The crash window is calculated as:
- `endDate = today − CRASH_DATE_OFFSET`
- `startDate = endDate − CRASH_NUM_DAYS + 1`

For example, `CRASH_DATE_OFFSET=4` and `CRASH_NUM_DAYS=1` with today = April 15 → window covers April 11.

**Zoho Projects severity/status IDs:**

Go to Zoho Projects portal → **Settings** → **Bug Tracker** → **Status** and **Severity** tabs. Each entry has a unique internal ID — use those for the `ZOHO_BUG_STATUS_*` and `ZOHO_BUG_SEVERITY_*` keys.

**Zoho Cliq webhook:**

Open Cliq → channel → **Integrations** → **Incoming Webhooks** → **Add webhook**. Copy the generated URL into `ZOHO_CLIQ_WEBHOOK_URL`.

**Config loading priority:**

`CRASH_ANALYSIS_PARENT` is the only environment variable the MCP client needs to provide — all other values are read from `crashpoint.config.json`. If the same key appears in both the JSON file and as an environment variable, **the environment variable wins**.

## 4. Run the Setup Command *(terminal bash script path only)*

> **⚡ MCP path users (Claude Desktop / Cursor): skip this section.** If you plan to use `run_full_pipeline` or `run_basic_pipeline` through Claude Desktop or Cursor, you do **not** need to run this command. The MCP tools detect an uninitialized workspace on their first call and run setup automatically — including generating `run_crash_pipeline.sh` with your real paths. See Section 5.

> **Terminal bash script path users: this step is required before you can run the bash pipeline for the first time.**

The bash pipeline script (`run_crash_pipeline.sh`) **does not exist** in your workspace until you run the setup command. The repo contains a **template** file (`automation/run_crash_pipeline.sh`) with placeholder values that must be replaced with your real paths. The setup command handles this automatically — you never need to manually edit any placeholders.

Run the setup command from the cloned repo directory:

```bash
CRASH_ANALYSIS_PARENT=/path/to/ParentHolderFolder node dist/cli.js setup
```

Replace `/path/to/ParentHolderFolder` with the absolute path to the folder you created in Section 3 (the value of `CRASH_ANALYSIS_PARENT` in your config).

**What the setup command does:**

- Creates the full folder structure:
  - `MainCrashLogsFolder/` (with `XCodeCrashLogs/`, `AppticsCrashLogs/`, `OtherCrashLogs/`)
  - `SymbolicatedCrashLogsFolder/`
  - `AnalyzedReportsFolder/`
  - `StateMaintenance/`
  - `Automation/`
- **Generates `Automation/run_crash_pipeline.sh`** — reads the template from the repo, replaces `<REPLACE_WITH_PATH_TO_PARENT_HOLDER_FOLDER>` with your `ParentHolderFolder` path and `<REPLACE_WITH_CRASHPOINT_PACKAGE_ROOT>` with the repo root path, and writes the result with correct executable permissions (0755)
- Copies `daily_crash_pipeline_prompt_phase1.md` and `daily_crash_pipeline_prompt_phase2.md` into `Automation/`
- Generates `.mcp.json` and the launchd plist for Claude CLI automation
- Creates `CurrentMasterLiveBranch` and `CurrentDevelopmentBranch` symlinks if `MASTER_BRANCH_PATH` / `DEV_BRANCH_PATH` are set in the config

After the setup command completes, the generated script at `ParentHolderFolder/Automation/run_crash_pipeline.sh` will contain your real paths — no placeholder text will remain.

> **Note:** You can also run this step through the MCP tool `setup_folders` in Claude Desktop or Cursor instead of the CLI command. Both do the same thing.

## 5. Register the MCP Server with Your AI Client

**Claude Desktop (macOS)** — edit `~/Library/Application Support/Claude/claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "crashpoint-ios": {
      "command": "npx",
      "args": ["-p", "github:maanasa-s-5539/CrashPoint-IOS-MCP", "crashpoint-ios-core"],
      "env": {
        "CRASH_ANALYSIS_PARENT": "/path/to/ParentHolderFolder"
      }
    }
  }
}
```

**Cursor** — add the same block to `.cursor/mcp.json` in your project root.

> **⚡ MCP path auto-setup:** Once your MCP client is configured, the **very first call** to `run_full_pipeline` or `run_basic_pipeline` will automatically create the entire workspace — all folders, `.mcp.json`, prompt templates, and `run_crash_pipeline.sh` with your real paths filled in. You do **not** need to run `node dist/cli.js setup` or `setup_folders` beforehand when using the MCP path.

> **Note:** The `.mcp.json` file generated inside `ParentHolderFolder` is used automatically by the Claude CLI automation script — you don't need to edit it.

## 6. Run the Bash Pipeline Script *(terminal bash script path only)*

> **⚡ MCP path users (Claude Desktop / Cursor):** You don't need to run this script manually. Just ask Claude to call `run_full_pipeline` — the workspace (including this script) is created automatically on the first call. See Section 5.

The bash pipeline script is generated by the setup command (Section 4) at `ParentHolderFolder/Automation/run_crash_pipeline.sh`. It:

1. Downloads crash data from Apptics and saves it locally.
2. Runs the full pipeline: export → symbolicate → analyze.
3. Sends a Zoho Cliq notification and creates/updates Zoho Projects bugs.
4. Reads the analysis report and source code to produce a `LatestFixPlan.md` with fix suggestions.

> **Note on the auto-setup block inside the script:** The generated script contains an auto-setup block (checking whether `StateMaintenance/` and `Automation/` exist). This is a **safety net for subsequent runs** — for example, if those folders were accidentally deleted. It is **not** the mechanism for first-time setup. You must run `node dist/cli.js setup` (Section 4) before the script exists and can be executed.

> **Re-running setup is safe** — it never overwrites existing files. To update automation files to the latest version from the repo template, run `setup_folders` with `force=true` or re-run `node dist/cli.js setup`.

**Run it manually from Terminal:**

```bash
# Foreground (output in terminal)
cd /path/to/ParentHolderFolder/Automation
bash run_crash_pipeline.sh

# Background with live log
bash /path/to/ParentHolderFolder/Automation/run_crash_pipeline.sh &
tail -f /path/to/ParentHolderFolder/Automation/ScheduledRunLogs/pipeline_*.log
```

Each run saves a timestamped log at `Automation/ScheduledRunLogs/pipeline_YYYY-MM-DD_HH-MM-SS.log`.

> **Warning:** If Phase 1 fails, Phase 2 is skipped entirely. Check the log for the specific error.

## 7. Scheduled Automation with macOS launchd

`setup_folders` generates a launchd plist at `~/Library/LaunchAgents/com.crashpipeline.daily_mcp.plist` that runs the script daily at the time set by `SCHEDULED_RUN_TIME`.

```bash
# Load (register) the job — run once after setup_folders
launchctl load ~/Library/LaunchAgents/com.crashpipeline.daily_mcp.plist

# Trigger immediately (for testing)
launchctl start com.crashpipeline.daily_mcp

# Unload (stop scheduled runs)
launchctl unload ~/Library/LaunchAgents/com.crashpipeline.daily_mcp.plist

# Check if loaded
launchctl list | grep crashpipeline
```

launchd stdout/stderr go to `/tmp/crashpipeline_stdout.log` and `/tmp/crashpipeline_stderr.log`.

> **Warning:** If the Mac is asleep at the scheduled time, the job runs when the Mac next wakes. To guarantee on-time execution:
> ```bash
> sudo pmset repeat wakeorpoweron MTWRFSU HH:MM:00
> ```
> Set the wake time a few minutes before `SCHEDULED_RUN_TIME`.

## 8. Claude CLI Setup (CRITICAL)

The automated bash pipeline calls the `claude` CLI in non-interactive mode (`claude -p`). Without proper CLI setup, Claude will either fail to find the MCP servers or pause waiting for permission confirmation that never arrives, hanging the pipeline.

### Step 1 — Install Claude CLI

Follow the [official Claude Code installation guide](https://docs.anthropic.com/en/docs/claude-code/getting-started). After installing, verify and record the binary path — you will need it for `CLAUDE_CLI_PATH` in `crashpoint.config.json`:

```bash
which claude
# Example output: /Users/yourname/.local/bin/claude
claude --version
```

### Step 2 — Add the Apptics MCP server

The CrashPoint MCP server is registered automatically via the `.mcp.json` generated by `setup_folders`. The Apptics MCP server must be added separately using `claude mcp add`:

```bash
# Add your Apptics MCP server (replace with your actual server name and command)
claude mcp add YOUR_APPTICS_MCP_NAME -- <command to start the Apptics MCP server>

# Confirm both servers are visible
claude mcp list
```

The name shown by `claude mcp list` for your Apptics server is the value to use for `APPTICS_MCP_NAME` in `crashpoint.config.json`.

### Step 3 — Configure `~/.claude/settings.json`

This file pre-authorizes the MCP servers, tools, and file paths so the automated pipeline can run without any prompts.

Create or edit `~/.claude/settings.json`:

```json
{
  "enabledMcpjsonServers": [
    "crashpoint-ios"
  ],
  "allowedTools": [
    "mcp__crashpoint-ios__*",
    "mcp__YOUR_APPTICS_MCP_NAME__*"
  ],
  "permissions": {
    "allow": [
      "Read",
      "Write",
      "Edit",
      "Glob",
      "Grep",
      "mcp__crashpoint-ios__*",
      "mcp__YOUR_APPTICS_MCP_NAME__*"
    ],
    "deny": []
  },
  "allowedPaths": [
    "/absolute/path/to/ParentHolderFolder",
    "/absolute/path/to/MasterBranch",
    "/absolute/path/to/DevBranch"
  ]
}
```

**What each field does:**

| Field | Purpose |
|---|---|
| `enabledMcpjsonServers` | Lists server names from `.mcp.json` in `ParentHolderFolder` that should be enabled. Add `"crashpoint-ios"` here. |
| `allowedTools` | MCP tool patterns that Claude may call without prompting. Use `mcp__<server-name>__*` to grant all tools for a server. |
| `permissions.allow` | Grants file operations (`Read`, `Write`, `Edit`, `Glob`, `Grep`) and MCP tool access without interactive confirmation. |
| `allowedPaths` | Absolute paths that Claude may read from and write to. Add `ParentHolderFolder`, `MASTER_BRANCH_PATH`, and `DEV_BRANCH_PATH`. |

**Fill in your actual paths:**

- `ParentHolderFolder` → value of `CRASH_ANALYSIS_PARENT` in your config
- `MasterBranch` → value of `MASTER_BRANCH_PATH` in your config
- `DevBranch` → value of `DEV_BRANCH_PATH` in your config
- `YOUR_APPTICS_MCP_NAME` → value of `APPTICS_MCP_NAME` in your config

> **Note:** If you are not using Apptics, omit the `mcp__YOUR_APPTICS_MCP_NAME__*` lines and the Apptics `allowedTools` entry.

Restart any running Claude CLI sessions after saving this file.

## 9. Verify the Pipeline

### MCP path (Claude Desktop / Cursor)

1. Call `run_full_pipeline` (or `run_basic_pipeline`) from Claude for the first time.
2. Confirm the workspace was auto-created: the folder structure should exist under `ParentHolderFolder` and `Automation/run_crash_pipeline.sh` should be present with your real paths (no `<REPLACE_WITH_...>` placeholders).
3. Place a test `.crash` file in `MainCrashLogsFolder/OtherCrashLogs/` and run the pipeline again to confirm symbolicated files appear in `SymbolicatedCrashLogsFolder/` and reports appear in `AnalyzedReportsFolder/`.
4. If `ZOHO_CLIQ_WEBHOOK_URL` is set, confirm a notification arrived in the Cliq channel.

### Terminal bash script path

1. After running `node dist/cli.js setup` (Section 4), confirm the folder structure exists under `ParentHolderFolder`.
2. **Confirm the setup command worked correctly:** Open `Automation/run_crash_pipeline.sh` and verify the `PARENT_HOLDER_FOLDER` line shows your real path (not `<REPLACE_WITH_...>`). This confirms the setup command successfully replaced the placeholders. You should never need to edit this file manually.
3. Place a test `.crash` file in `MainCrashLogsFolder/OtherCrashLogs/`.
4. Run the CLI pipeline:
   ```bash
   CRASH_ANALYSIS_PARENT=/path/to/ParentHolderFolder node dist/cli.js pipeline
   ```
5. Confirm symbolicated files appear in `SymbolicatedCrashLogsFolder/` and reports appear in `AnalyzedReportsFolder/`.
6. Run the bash script manually and check the log:
   ```bash
   bash /path/to/ParentHolderFolder/Automation/run_crash_pipeline.sh
   cat /path/to/ParentHolderFolder/Automation/ScheduledRunLogs/pipeline_*.log
   ```
7. If `ZOHO_CLIQ_WEBHOOK_URL` is set, confirm a notification arrived in the Cliq channel.

## 10. Troubleshooting

| Problem | Fix |
|---|---|
| `CRASH_ANALYSIS_PARENT is required` | Add it to the `env` block in your MCP client config. Use an absolute path with no trailing slash. |
| `symbolicatecrash not found` | Run `xcode-select --install`. If multiple Xcodes exist, set the active one: `sudo xcode-select -s /Applications/Xcode.app/Contents/Developer` |
| dSYM UUID mismatch | The `.dSYM` doesn't match the crashing build. Run `node dist/cli.js verify-dsym` to compare UUIDs. Use the `.dSYM` from the same archive as the build. |
| `jq is required but not found` | `brew install jq` |
| Claude CLI hangs during automation | Missing entries in `~/.claude/settings.json`. Verify `allowedPaths` includes `ParentHolderFolder`, `MASTER_BRANCH_PATH`, and `DEV_BRANCH_PATH`, and that both MCP server names appear in `allowedTools` and `permissions.allow`. See Section 8. |
| Phase 1 fails, Phase 2 skipped | Check the log file. Common causes: wrong Apptics credentials, network unavailable, or missing config values. |
| launchd job doesn't run | Check `launchctl list \| grep crashpipeline`. If not loaded, run the `launchctl load` command. Check `/tmp/crashpipeline_stdout.log` and `/tmp/crashpipeline_stderr.log`. |
| Invalid JSON in config | Validate: `node -e "JSON.parse(require('fs').readFileSync('/path/to/crashpoint.config.json','utf8'))"` |
